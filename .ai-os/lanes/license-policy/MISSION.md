# 证道 Mission

> 当前 lane 记录的是“仓库开源许可证收紧与授权口径统一”这条交付线。本轮目标是在保持开源属性前提下，把仓库顶层授权从宽松许可切换为强 copyleft 许可，并同步所有对外元数据。

## 1. 交付基线摘要

- **宿主项目 / 系统**：证道 Electron 桌面端创作软件
- **当前交付主题**：开源许可证收紧与仓库授权口径统一
- **当前交付目标**：把仓库顶层授权从 `MIT` 调整为 `AGPL-3.0-only`，同步 `LICENSE`、README 与包元数据，避免仓库对外授权口径不一致
- **成功标准**：
  - 顶层 `LICENSE` 替换为 GNU AGPL v3 正式文本
  - `README.md` 与 `package.json` 的许可证说明统一为 `AGPL-3.0-only`
  - `package-lock.json` 根包许可证字段与 `package.json` 保持一致
  - 对外文档明确这是强 copyleft 开源协议，而不是宽松协议
- **项目模式**：change
- **当前交付档位**：license-policy
- **当前治理档位**：P1
- **当前基线 ID**：CR-20260421-225658-oss-license-tightening

## 2. 用户与闭环场景

- **目标用户**：
  - 仓库维护者
  - 使用、分发、修改或二次开发证道的外部开发者
- **关键场景**：
  - 用户打开仓库首页时，能明确看到项目不再是 MIT，而是强 copyleft 开源许可
  - 包管理与仓库元数据对许可证的声明保持一致
  - 维护者在后续发版时不会继续沿用旧的 MIT 授权口径
- **当前最小可行闭环**：顶层许可证文本 + README 许可说明 + npm 元数据统一
- **明确后续迭代项**：
  - 是否追加每个源文件头部版权/许可声明
  - 是否补充贡献者入站授权说明
  - 是否提供双许可或商业许可

## 3. 已确认约束与关键决策

- **已确认技术栈与关键选型**：
  - 仓库当前使用 Node/npm 元数据描述许可证
  - 本轮只处理仓库自身授权，不改第三方依赖许可证
- **必须保持的宿主项目约束**：
  - 不改业务逻辑、不改发布产物行为
  - 不重写历史版本的既有发布事实
  - 不在未确认前擅自引入“非开源” source-available 许可
- **已确认核心决策**：
  - 采用 `AGPL-3.0-only`，而不是 `AGPL-3.0-or-later`
  - 本轮不做双许可
  - 本轮不批量补所有源码文件头声明

## 4. 范围边界与非目标

### 范围内
- 顶层 `LICENSE`
- `README.md` 的许可说明
- `package.json` / `package-lock.json` 的根包许可证字段
- 变更基线、验收与发布说明

### 范围外
- 第三方依赖许可证扫描或替换
- 历史 release 重新打标或补发法律说明
- 商业许可文本
- CLA / DCO 流程

### 非目标
- 不把项目改成“禁止商用”的非开源协议
- 不顺手修改无关开源治理文档
- 不在本轮给所有源文件逐个加许可证头

## 5. 稳定风险与外部依赖

- **外部依赖**：GNU AGPL v3 正式许可文本、OSI 对 AGPL 的开源认可、npm SPDX 标识
- **稳定风险**：
  - 对外协作者如果仍按 MIT 理解权限，会产生授权误解
  - 若用户真正想要的是“更严格但限制商用/SaaS”的协议，AGPL 可能仍不够“严格”
  - 许可证切换不会自动改变历史已发布版本的既有授权事实
- **高风险触发因素**：误把 AGPL 描述成“禁止商用”、误写为非 SPDX 标识、漏改 README/package 元数据
- **影响范围摘要**：仓库首页授权说明、npm 元数据、后续 release 口径
- **审批点**：已于 2026-04-21 确认按 `AGPL-3.0-only` 执行
